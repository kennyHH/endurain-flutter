# Endurain Mobile v0.0.34 – Feature Overview

## Release focus

Version 0.0.34 focuses on reducing first-run confusion during tracking start, hardening server-side activity deletion reliability, and improving retry upload UX in Activity Details.

## What users will notice

### 1) Tracking start now has a clear first-run permission gate

When a user taps `Start tracking` without active location permission, the app now opens the setup flow directly before starting.

This removes the impression that tracking is broken and gives users an immediate path to enable required permissions.

The onboarding trigger is no longer dependent on server login state, so first-run guidance appears consistently after unlock on fresh installs.

### 2) Permission onboarding stays pending until access is really granted

If location permission is still denied after setup, onboarding is no longer treated as complete.

Users continue to get guided setup instead of being left without prompts.

### 3) Activity delete flow is more robust against server mismatch

Delete requests now prefer persisted server-side activity IDs and avoid sending overflow-prone local timestamp IDs.

This improves reliability for delete-on-server operations and prevents ambiguous backend failures.

### 4) Retry Upload action now clears immediately after success

After a successful retry upload in Activity Details, the `Retry Upload` button now disappears immediately in place.

Users no longer need to leave and reopen details to see the updated uploaded state.

### 5) GPX export shares with naming convention

Activity Details exports GPX via share flow using a real file with the naming-convention filename.

Shared files use the naming convention (`YYYY-MM-DD_HH-mm_Type[_Name].gpx`) consistently.

### 6) Pace appears earlier on very short runs

For short activities, pace chart sampling starts earlier so the first visible pace is no longer delayed until around 10 meters.

### 7) Average speed now shows in run/walk details

Activity Details now includes average speed as an explicit metric for run and walk sessions, while keeping pace visible.

### 8) Minimal start-window tuning for first meters

For run/walk sessions, the first ~8 seconds now accept high-confidence short segments around 4 meters more reliably.

This improves early-distance continuity without loosening broader accuracy/speed guardrails for the full activity.

### 9) Pace display now follows server-compatible geometry

Run/walk pace display now prefers track-geometry-based distance for pace calculations, matching import-side server behavior more closely.

### 10) Session refresh now runs proactively and safely

The app now triggers silent token refresh globally on app resume and in periodic lifecycle intervals.

Refresh runs only when tokens are near expiry and concurrent refresh attempts are deduplicated into a single in-flight operation.

### 11) Empty or meaningless activities are now blocked before upload

Activities that are too short, have insufficient GPS points, or look like stationary jitter are now marked as non-uploadable.

Retry upload is replaced by a clear hint in Activity Details to prevent endless retry loops against server endpoints.

### 12) History metrics now hydrate from full activities

When lightweight History summaries miss track-backed metrics, cards now hydrate from full activity rows so pace/elevation align better with Activity Details.

### 13) Upload now persists server metrics locally

After successful uploads, server-returned distance, duration, and elevation values are persisted in local activity records for closer parity with server views.

### 14) History now rehydrates implausible summary cards

If a History summary row includes preview track points but still reports implausible metrics (such as 0s duration or 0.00km distance), the app now reloads full activity data before rendering card metrics.

Additionally, History now retries hydration on a timed cadence while summary rows remain implausible, preventing stale zero-value cards from persisting after an activity is completed but still pending upload.

## Technical scope (for maintainers)

### Tracking start UX

- Added a start-time permission gate in `MapScreen` before tracking begin.
- Kept permission setup entrypoint consistent with existing onboarding dialog flow.

### Permission onboarding state

- `runPermissionOnboardingSetup` now marks onboarding as completed only when location permission is actually granted.
- On denied permission, onboarding remains pending for future guided retries.

### Upload/delete reliability

- Persisted and reused `server_activity_id` for delete requests.
- Added numeric range guard for backend-compatible delete IDs.
- Preserved clearer non-405 delete error details over fallback 405 noise.
- Updated Activity Details retry flow to set uploaded state immediately after successful retry callback.
- Updated GPX export to share a real file path with convention-based filename generation.
- Adjusted short-distance pace window behavior to emit early valid pace points for tiny tracks.
- Added average speed metric tile for non-ride activity details.
- Tuned warmup min-distance thresholds for early high-confidence run/walk samples.
- Aligned run/walk pace display with server-compatible track geometry calculations.
- Added app-wide proactive token refresh triggers and token-fresh gating.
- Added single-flight refresh deduplication to prevent parallel token refresh races.
- Added upload eligibility matrix for short/empty/jitter activities and blocked retry CTA for non-uploadable details.
- Hydrated History card metrics from full activities when summary rows are incomplete.
- Persisted server-returned distance/duration/elevation metrics after successful activity upload.
- Added guardrails to hydrate full activity metrics when summary rows contain route points but implausible duration/distance.
- Added timed stale-snapshot re-hydration for implausible pending-upload history summaries.

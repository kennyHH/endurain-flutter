# Endurain Mobile v0.0.33 – Feature Overview

## Release focus

Version 0.0.33 focuses on upload reliability, clearer GPS start readiness, and cleaner file/export behavior for support and self-hosted server workflows.

## What users will notice

### 1) Clearer GPX export filenames

GPX exports and upload payload filenames now use a readable, consistent format:

- `YYYY-MM-DD_HH-mm_Type.gpx`
- `YYYY-MM-DD_HH-mm_Type_Name.gpx` (when an activity name exists)

This improves:

- file management in Downloads/cloud folders,
- traceability during support cases,
- consistency between share/export and upload flows.

### 2) Better tracking start behavior with GPS readiness

Before a stable GPS fix is available, the main `Start tracking` control stays visible but disabled, with explicit guidance.

This removes the “tap does nothing” impression and makes readiness status obvious without hiding the primary action.

### 3) Upload visibility compatibility with Endurain servers

Upload requests now send explicit visibility metadata so server-side defaults are less likely to mark imports as hidden.

The upload flow now includes visibility-compatible multipart fields and fallback variants for broader backend compatibility.

### 4) Stronger protection against duplicate uploads

v0.0.33 introduces multiple layers that work together:

- in-flight deduplication for concurrent uploads of the same activity,
- idempotency headers per activity upload request,
- persisted local upload state (`uploaded=true`) after successful upload,
- pre-upload short-circuit when an activity is already marked uploaded,
- queue replay persistence so successful retries do not get re-uploaded later.

Result: fewer accidental duplicate activities on the server, especially in retry/reconnect scenarios.

## Technical scope (for maintainers)

### Upload pipeline

- Multipart upload metadata expanded for visibility handling.
- Stable request identity headers included (`Idempotency-Key`, `X-Upload-Activity-Id`).
- Concurrent request deduplication per `activity.id`.
- Repository-backed upload-success persistence and upload-guard checks.

### Tracking controls

- Start action disabled until GPS readiness criteria are met.
- User-facing readiness hint shown near start control.
- Countdown/start status messaging refined for better transparency.

### GPX handling

- Unified filename builder used across export and upload paths.
- Sanitization and truncation behavior retained while improving readability.

## Quality and validation

The release candidate has been validated with:

- static analysis (`flutter analyze`),
- automated tests (`flutter test`),
- upload service regression coverage for:
  - multipart metadata,
  - in-flight deduplication,
  - already-uploaded short-circuit,
  - queue success persistence.
